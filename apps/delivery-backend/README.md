## Overview

NestJS 11 service that exposes:

- A **REST API** (`/api/orders`) to create and manage delivery orders persisted in **MongoDB** via Mongoose.
- A **Socket.IO WebSocket Gateway** (`/orders` namespace) that broadcasts real-time events to connected clients every time an order is created or its status changes.

## Architecture

```
┌──────────────────────────────────────────────┐
│                  AppModule                   │
│  ┌─────────────┐   ┌──────────────────────┐  │
│  │ ConfigModule│   │   MongooseModule     │  │
│  │  (global)   │   │  (MongoDB via URI)   │  │
│  └─────────────┘   └──────────────────────┘  │
│                                              │
│  ┌───────────────────────────────────────┐   │
│  │           OrdersModule                │   │
│  │                                       │   │
│  │  OrdersController  ──► OrdersService  │   │
│  │        │                    │         │   │
│  │        │             Mongoose Model   │   │
│  │        │                             │   │
│  │  OrdersGateway ◄───────────────────── │   │
│  │  (Socket.IO /orders)                  │   │
│  └───────────────────────────────────────┘   │
└──────────────────────────────────────────────┘
```

**Design decisions:**
- `OrdersController` handles HTTP and delegates events to `OrdersGateway` directly — keeps the service layer pure (no WS coupling).
- `OrdersService` always returns plain `IOrder` objects, never raw Mongoose documents — controllers and gateways are decoupled from the ODM.
- `ConfigModule` is global; all env vars are loaded once through a typed factory (`src/config/configuration.ts`).
- A global `ValidationPipe` with `whitelist: true` and `forbidNonWhitelisted: true` rejects any request with undeclared properties.

## Data Model

MongoDB collection: **`orders`**

| Field             | Type                                    | Notes                                              |
|-------------------|-----------------------------------------|----------------------------------------------------|
| `_id`             | `ObjectId`                              | Auto-generated by MongoDB                          |
| `customerName`    | `string`                                | **Required** · max 120 chars                       |
| `deliveryAddress` | `string`                                | **Required** · max 255 chars                       |
| `status`          | `"pending" \| "in_transit" \| "delivered"` | Default: `"pending"` · indexed                  |
| `coordinates`     | `{ lat: number, lng: number } \| null`  | Simulated CDMX coords if not provided              |
| `createdAt`       | `Date`                                  | Auto-managed by Mongoose `timestamps: true`        |
| `updatedAt`       | `Date`                                  | Auto-managed by Mongoose `timestamps: true`        |

### `OrderStatus` enum

```typescript
enum OrderStatus {
  PENDING    = 'pending',
  IN_TRANSIT = 'in_transit',
  DELIVERED  = 'delivered',
}
```

## REST API

Base URL: `http://localhost:3001/api`

### `POST /api/orders`

Create a new delivery order.

**Request body**

```jsonc
{
  "customerName":    "María García",       // required
  "deliveryAddress": "Insurgentes Sur 1234, CDMX", // required
  "coordinates": {                          // optional
    "lat": 19.4284,
    "lng": -99.1277
  }
}
```

**Response `201`**

```jsonc
{
  "_id":             "6655aabbccddeeff00112233",
  "customerName":    "María García",
  "deliveryAddress": "Insurgentes Sur 1234, CDMX",
  "status":          "pending",
  "coordinates":     { "lat": 19.4284, "lng": -99.1277 },
  "createdAt":       "2026-05-27T12:00:00.000Z",
  "updatedAt":       "2026-05-27T12:00:00.000Z"
}
```

**Validation errors `400`** — if `customerName` or `deliveryAddress` are missing or blank.

---

### `GET /api/orders`

List all orders sorted by `createdAt` descending.

**Response `200`** — array of order objects.

---

### `GET /api/orders/:id`

Fetch a single order by MongoDB ObjectId.

**Response `200`** — order object.  
**Response `404`** — `{ "message": "Order with id \"...\" was not found" }`.

---

### `PATCH /api/orders/:id/status`

Update an order's status. Triggers a `order:status_updated` WebSocket broadcast.

**Request body**

```jsonc
{
  "status": "in_transit"  // "pending" | "in_transit" | "delivered"
}
```

**Response `200`** — updated order object.  
**Response `404`** — order not found.  
**Response `400`** — invalid status value.

## WebSocket Gateway

**URL:** `ws://localhost:3001/orders`  
**Namespace:** `/orders`  
**Transport:** WebSocket + polling fallback (Socket.IO)

### Connecting (JavaScript / React Native)

```typescript
import { io } from 'socket.io-client';

const socket = io('http://localhost:3001/orders', {
  transports: ['websocket'],
});

socket.on('connect', () => console.log('Connected:', socket.id));
```

### Events emitted by the server

| Event                  | Trigger                              | Payload    |
|------------------------|--------------------------------------|------------|
| `order:created`        | `POST /api/orders` succeeds          | `IOrder`   |
| `order:status_updated` | `PATCH /api/orders/:id/status` succeeds | `IOrder` |

### Event payload shape (`IOrder`)

```typescript
interface IOrder {
  _id:             string;
  customerName:    string;
  deliveryAddress: string;
  status:          'pending' | 'in_transit' | 'delivered';
  coordinates:     { lat: number; lng: number } | null;
  createdAt:       string; // ISO 8601
  updatedAt:       string; // ISO 8601
}
```

## Environment Variables

Copy `.env.example` to `.env` and adjust values.

| Variable       | Default                                      | Description                              |
|----------------|----------------------------------------------|------------------------------------------|
| `NODE_ENV`     | `development`                                | Node environment                         |
| `PORT`         | `3001`                                       | HTTP + WS server port                    |
| `MONGODB_URI`  | `mongodb://localhost:27017/delivery_db`      | Mongoose connection string               |
| `REDIS_HOST`   | `localhost`                                  | Redis host (future pub/sub adapter)      |
| `REDIS_PORT`   | `6379`                                       | Redis port                               |
| `REDIS_PASSWORD` | —                                          | Redis password                           |
| `CORS_ORIGIN`  | `http://localhost:8081`                      | Comma-separated allowed origins          |
| `WS_NAMESPACE` | `/orders`                                    | Socket.IO namespace for order events     |

> **Note:** when running inside Docker the `MONGODB_URI`, `REDIS_HOST`, and `REDIS_PASSWORD` values are overridden by `docker-compose.yml` to use internal service names.

## Getting Started

### Prerequisites

- Node.js ≥ 18
- MongoDB 7 running locally (or use Docker — see below)

### Local development

```bash
# 1. Install dependencies
npm install

# 2. Copy and edit environment file
cp .env.example .env

# 3. Start in watch mode
npm run start:dev
```

The server will be available at:
- REST  → `http://localhost:3001/api`
- WS    → `ws://localhost:3001/orders`

## Running with Docker

Start the entire stack (MongoDB + Redis + backend) from the **monorepo root**:

```bash
# Build and start all services
docker compose up -d

# Stream backend logs
docker compose logs -f delivery-backend

# Hot-reload source files without rebuilding the image
docker compose watch
```

The `docker-compose.yml` automatically injects the correct `MONGODB_URI` and `REDIS_*` values for inter-container networking.

## Testing

### Unit tests

```bash
npm run test
```

Tests live next to their source files (`*.spec.ts`).  
The `OrdersService` suite covers: `create`, `findAll`, `findOne`, `updateStatus` including 404 paths — all using a mocked Mongoose model (no DB required).

### E2E / Integration tests

```bash
# Requires a running MongoDB instance (use Docker)
npm run test:e2e
```

The E2E suite (`test/app.e2e-spec.ts`) boots a real NestJS application and:

1. Validates that `POST /api/orders` rejects missing required fields.
2. Runs the **full real-time flow**:
   - Creates an order via HTTP.
   - Asserts the `order:created` Socket.IO event arrives before the HTTP response is checked.
   - Updates the order status via HTTP.
   - Asserts the `order:status_updated` Socket.IO event arrives with the new status.
3. Checks list and not-found endpoints.

### Coverage

```bash
npm run test:cov
```
